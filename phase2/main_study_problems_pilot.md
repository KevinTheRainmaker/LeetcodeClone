# 메인 스터디 후보 문제 명세 및 One-shot 파일럿 프로토콜

> 모든 공개 테스트의 기댓값은 동봉된 레퍼런스 솔루션(`sol_a1~a3.py`, `sol_b1~b3.py`)으로 생성·검산됨.
> ⚠️ 명세 중 별표(★)가 붙은 규칙 정의는 명세 완결성을 위해 임의로 확정한 것이므로 사용 전 검토 필요.
>
> **사용 순서**: one-shot 파일럿은 각 패밀리 대표 1문제(A1, B1)로 수행 → 패밀리 채택 후 해당 변형 2개(A2·A3 또는 B2·B3)를 본 실험에 투입. 변형들은 동일 골격(A: 격자 + 픽업/배달 미션 + 특수 메커니즘 1개 / B: 요청 스트림 + 결정적 처리 규칙 + 개체별 출력)에 메커니즘만 교체한 동형 설계이며, 난이도 동등성은 사람 시간 파일럿으로 별도 확인 필요.

## 패밀리 A — 문제 A1: 배달 로봇 (격자 시뮬레이션 / 확장 상태 탐색)

### 문제 설명

N×M 격자 창고에서 배달 로봇이 물건을 픽업해 배달 지점까지 운반한다.
미션 완료까지 걸리는 **최소 틱(tick) 수**를 구하라. 불가능하면 `-1`을 출력한다.

격자 기호:

| 기호 | 의미              |
| ---- | ----------------- |
| `.`  | 빈 칸 (통행 가능) |
| `#`  | 벽 (통행 불가)    |
| `S`  | 로봇 시작 위치    |
| `P`  | 픽업 지점         |
| `D`  | 배달 지점         |
| `C`  | 충전소            |

`S`, `P`, `D`, `C`는 모두 통행 가능한 칸이며, 격자에 각각 정확히 1개 존재한다(`C`는 0개 이상).

**행동 규칙** — 매 틱마다 로봇은 다음 중 정확히 하나를 수행한다:

1. **이동**: 상·하·좌·우 인접 칸으로 1칸 이동. 벽과 격자 밖으로는 불가.
2. **대기**: 제자리에 머무름.

**배터리 규칙**:

- 시작 배터리와 최대 용량은 모두 `B`.
- 이동 비용: 비적재 시 1, **적재 시 2**.
- 이동에 필요한 배터리가 부족하면 그 이동은 불가. 이동 후 배터리가 정확히 0이 되는 것은 허용. ★
- **충전소(`C`) 칸에서 대기**하면 틱당 +2 충전 (최대 `B`까지). 충전소를 지나치기만 하면 충전되지 않는다. ★ 충전소가 아닌 칸에서의 대기는 배터리 변화 없음.

**픽업/배달 규칙**:

- `P` 칸에 위치하게 되는 즉시 자동으로 픽업된다 (별도 틱 소모 없음, 1회만). ★
- **적재 상태로 `D` 칸에 도착하는 순간의 틱 수**가 답이다.

### 입력 형식

```
N M B
(N줄의 격자, 각 줄 M글자)
```

제약: 1 ≤ N, M ≤ 20, 1 ≤ B ≤ 50

### 출력 형식

최소 틱 수 (정수), 불가능하면 `-1`

### 공개 테스트 케이스

| ID  | 의도                                    | 입력                                  | 정답 |
| --- | --------------------------------------- | ------------------------------------- | ---- |
| T1  | 기본 동작                               | `1 7 10` / `S..P..D`                  | `6`  |
| T2  | 적재 비용 2배 → 사전 충전 필요          | `1 7 8` / `S.CP..D`                   | `7`  |
| T3  | 배터리 정확히 0으로 도착 (허용 경계)    | `1 5 6` / `S.P.D`                     | `4`  |
| T4  | 불가능 판정                             | `1 5 5` / `S.P.D`                     | `-1` |
| T5  | 경유 충전 + 충전량 최적화 + 0 도착 복합 | `1 9 8` / `S..C..P.D`                 | `9`  |
| T6  | 벽 우회 + 경유 충전                     | `3 5 9` / `S.#.D` / `..#P.` / `..C..` | `9`  |

### 히든 테스트 설계 지침 (결과 채점용)

- 충전소 2개 이상, 어느 쪽 경유가 최적인지 비자명한 배치
- 픽업 **후** 충전소로 회귀하는 게 최적인 케이스 (적재 비용 2배 상태에서의 충전 판단)
- `P`가 `D`보다 멀리 있어 왕복 구조가 되는 격자
- B가 큰데도 벽 구조 때문에 `-1`인 케이스 (배터리가 아닌 연결성 실패)
- 대기가 충전소 밖에서는 절대 이득이 아님을 확인하는 케이스

### LLM의 예상 실패 지점 (실패의 "질" 판정 기준)

규칙 상호작용 실패로 **인정**되는 것:

- 상태에 적재 여부/배터리를 포함하지 않은 단순 BFS (visited를 위치로만 관리)
- 적재 시 비용 2배를 경로 후반에 미반영
- 충전을 "충전소 통과"로 구현 (대기 행동 누락)
- 충전 상한 `B` 미처리, 정확히-0 경계를 `< 0`/`<= 0`으로 혼동

명세 표기 문제로 **분류**되는 것 (난이도가 아니라 명세를 고칠 신호):

- 입력 파싱 오류, 출력 형식 불일치, 기호 해석 오류

## 패밀리 A — 문제 A2: 보안 구역 로봇 (주기 개폐 문)

### 문제 설명

A1과 동일한 픽업(`P`)→배달(`D`) 미션. 배터리 대신 **주기적으로 여닫히는 보안문**이 특수 메커니즘이다. 미션 완료까지 최소 틱 수, 불가능하면 `-1`.

격자 기호: `.` `#` `S` `P` `D`는 A1과 동일, `G` = 보안문.

**문 규칙**:

- 모든 문은 동일한 주기를 공유한다: 틱 0부터 **O틱 동안 열림 → C틱 동안 닫힘**을 반복. 즉 시각 t에 문이 열려 있을 조건은 `t mod (O+C) < O`.
- 로봇은 **어떤 시각에든 닫힌 문 칸 위에 존재할 수 없다.** ★ 즉, 문 칸으로 이동하려면 **도착 시각**에 열려 있어야 하고, 문 칸 위에서 대기하려면 **다음 시각**에도 열려 있어야 한다.

**행동 규칙**: 매 틱 이동(상하좌우 1칸) 또는 대기. 배터리 없음. 픽업·완료 규칙은 A1과 동일 (★ 포함).

### 입력 형식

```
N M O C
(N줄의 격자)
```

제약: 1 ≤ N, M ≤ 20, 1 ≤ O, C ≤ 10. `S`는 문 칸이 아니다.

### 공개 테스트 케이스

| ID  | 의도                           | 입력                                    | 정답 |
| --- | ------------------------------ | --------------------------------------- | ---- |
| T1  | 문 없는 기본 동작              | `1 5 1 1` / `S.P.D`                     | `4`  |
| T2  | 문 앞 대기 후 통과             | `1 4 1 2` / `SGPD`                      | `5`  |
| T3  | 연속 문 2개 — 출발 위상 조절   | `1 6 2 1` / `SGGP.D`                    | `7`  |
| T4  | 연속 문 2개 + O=1 → 통과 불가  | `1 5 1 1` / `SGGPD`                     | `-1` |
| T5  | P–D 사이의 문 (적재 후 대기)   | `1 5 2 2` / `SGPGD`                     | `5`  |
| T6  | 우회 경로 vs 대기 트레이드오프 | `3 5 1 3` / `SG..D` / `.#.#.` / `..GP.` | `8`  |

### 히든 테스트 설계 지침

- 위상이 어긋난 두 경로 중 짧은 쪽이 대기 때문에 더 느려지는 격자
- 문 위에서의 대기가 닫힘 직전 1틱만 허용되는 경계 케이스
- O+C가 크고 출발 직후 장시간 대기가 최적인 케이스
- 문이 전혀 없는 큰 격자 (기본 BFS 회귀 확인)

### LLM의 예상 실패 지점

규칙 상호작용 실패로 **인정**: visited를 위치만으로 관리 (시간 위상 차원 누락 → 오답 또는 대기 경로 탐색 불가) / 문 판정을 도착 시각이 아닌 출발 시각으로 / 대기 행동 자체를 미구현 / 닫힌 문 위 대기를 허용.
명세 표기 문제로 **분류**: 파싱, `O`/`C` 순서 혼동, 출력 형식.

## 패밀리 A — 문제 A3: 빙판 창고 로봇 (미끄러짐 이동)

### 문제 설명

A1과 동일한 픽업(`P`)→배달(`D`) 미션. 특수 메커니즘은 **빙판 미끄러짐**: 로봇은 방향을 선택하면 멈출 때까지 미끄러진다. 미션 완료까지 **최소 이동 횟수**(미끄러짐 1회 = 1틱), 불가능하면 `-1`.

격자 기호: `.` `#` `S` `P` `D`는 동일, `R` = 거친 바닥.

**이동 규칙**:

- 매 틱 상·하·좌·우 중 한 방향을 선택하면, 로봇은 그 방향으로 **벽(`#`) 또는 격자 경계 직전 칸까지 미끄러진다.**
- 단, 미끄러지는 도중 **거친 바닥(`R`)에 진입하면 그 칸에서 즉시 정지**한다.
- 선택한 방향의 바로 옆 칸이 벽/경계면 그 이동은 불가능하다 (틱 소모 없음). ★
- 대기 행동은 없다.

**픽업/완료 규칙**:

- `P`는 **미끄러져 지나치기만 해도** 픽업된다 (정지 불필요). ★
- 완료는 적재 상태로 `D`에 **정지**해야 인정된다. 미끄러져 지나치는 것은 무효. ★ (픽업과 배달의 비대칭에 주의)

### 입력 형식

```
N M
(N줄의 격자)
```

제약: 1 ≤ N, M ≤ 20

### 공개 테스트 케이스

| ID  | 의도                                | 입력                                                              | 정답 |
| --- | ----------------------------------- | ----------------------------------------------------------------- | ---- |
| T1  | 기본: 벽 제동 3연속                 | `4 4` / `S..#` / `..P#` / `...#` / `#D.#`                         | `3`  |
| T2  | D 지나침 함정 — 측면 벽 접근 필요   | `4 6` / `S.P..#` / `.#....` / `..#D..` / `....#.`                 | `3`  |
| T3  | 미끄러지며 픽업 (정지 없이)         | `2 6` / `S..P.#` / `#...D#`                                       | `2`  |
| T4  | D 사방 너머가 개방 → 정지 불가      | `3 5` / `S.P..` / `..D..` / `.....`                               | `-1` |
| T5  | R 제동이 유일한 경로 (R 제거 시 -1) | `5 7` / `S..#...` / `.......` / `..R.P.D` / `.......` / `.......` | `3`  |

### 히든 테스트 설계 지침

- D에 적재 없이 먼저 정지한 뒤 P를 회수하고 재접근하는 왕복 구조
- R 위에서 출발하는 미끄러짐 (R은 진입 시에만 제동 ★ — 출발 칸으로서는 일반 칸)
- P를 두 번 지나는 경로 (픽업 1회 처리 확인)
- 최단 경로가 D를 두 번 통과한 뒤에야 정지하는 케이스

### LLM의 예상 실패 지점

규칙 상호작용 실패로 **인정**: 미끄러짐을 1칸 이동으로 구현 / D 통과를 완료로 처리 (비대칭 규칙 위반 — 가장 빈발 예상) / 통과 픽업 누락 (P 정지 요구) / R 제동 미구현 또는 R에서 정지 후 재출발 처리 오류.
명세 표기 문제로 **분류**: 파싱, 격자 좌표계 혼동.

## 패밀리 B — 문제 B1: 주차장 정산 시스템 (이벤트 기반 규칙 시뮬레이션)

### 문제 설명

주차장의 입·출차 이벤트 기록과 정산 시각이 주어진다. 차량별 총 주차 요금을 계산하라.

**요금 규칙**:

1. **기본요금**: 입차 1건당 1,000원 (최초 30분 포함). 주차 시간이 30분 이하여도 부과.
2. **추가요금**: 30분 초과분에 대해 **10분 블록 단위로 올림**, 블록당 500원.
   - 블록 i의 시작 시각 = 입차 시각 + 30분 + 10×i분.
3. **야간 할인**: 블록의 **시작 시각**이 야간(22:00 이상 또는 08:00 미만)이면 그 블록은 500원 대신 250원. ★ 기본요금에는 야간 할인이 없다.
4. **일 상한**: 차량별로 **역일(calendar day)별 귀속 요금 합**에 15,000원 상한 적용.
   - 기본요금은 입차 시각이 속한 날에 귀속. ★
   - 각 블록은 그 블록의 시작 시각이 속한 날에 귀속. ★
5. **경차 할인**: 차종이 `COMPACT`인 차량은 일 상한 적용 **후** 전체 합계의 50% (원 미만 내림). ★ 적용 순서: 야간 할인 → 일 상한 → 경차 할인.
6. **다회 입출차**: 같은 차량의 여러 세션은 각각 계산 후 합산 (단, 일 상한은 역일별 누적에 적용).
7. **미출차 처리**: 정산 시각까지 출차 기록이 없는 차량은 **정산 시각에 출차한 것으로 간주**. ★

### 입력 형식

```
E (정산일) (정산시각 HH:MM)
(E줄의 이벤트: day HH:MM 차량번호 차종 IN|OUT)
```

- `day`는 1부터 시작하는 일 번호, 차종은 `NORMAL` 또는 `COMPACT`
- 이벤트는 시각 오름차순으로 주어지며, IN/OUT 쌍은 항상 정합적이다.

### 출력 형식

차량번호 오름차순으로 한 줄에 `차량번호 요금`

### 공개 테스트 케이스

| ID  | 의도                                 | 입력 (요약)                                       | 정답                      |
| --- | ------------------------------------ | ------------------------------------------------- | ------------------------- |
| T1  | 기본 (주간 75분)                     | `1234` 10:00 IN → 11:15 OUT                       | `1234 3500`               |
| T2  | 야간 경계 걸침 (블록 시작 기준 판정) | `5566` 21:00 IN → 22:25 OUT                       | `5566 3250`               |
| T3  | 일 상한 도달                         | `7777` 06:00 IN → 23:00 OUT                       | `7777 15000`              |
| T4  | 자정 걸침 (블록의 역일 분리 귀속)    | `2222` day1 23:00 IN → day2 01:00 OUT             | `2222 3250`               |
| T5  | 경차 할인 (내림 포함)                | T1과 동일, `COMPACT`                              | `3333 1750`               |
| T6  | 다회 입출차 + 미출차 간주            | `1111` 2세션 + `9999` 21:50 IN 미출차, 정산 23:00 | `1111 3000` / `9999 1000` |

(전체 입력 원문은 동봉된 `tests_b1.py` 참조)

### 히든 테스트 설계 지침

- 정확히 30분 주차 (추가요금 0 경계)
- 블록 시작이 정확히 22:00 / 08:00 (야간 판정 경계)
- 자정 걸침 + 양일 모두 상한 도달
- 경차 + 일 상한 동시 발동 (할인 순서 검증)
- 입차 직후 출차 (0분, 기본요금만)
- 같은 차량이 같은 날 3세션 이상 (역일 누적 상한)

### LLM의 예상 실패 지점

규칙 상호작용 실패로 **인정**되는 것:

- 야간 판정을 블록 시작이 아닌 입차/출차 시각 기준으로 구현
- 일 상한을 역일이 아닌 세션 단위에 적용
- 블록의 역일 귀속 누락 (자정 걸침 케이스 실패)
- 할인 적용 순서 오류 (경차 할인을 상한 전에 적용)
- 올림/내림 처리 오류

명세 표기 문제로 **분류**되는 것:

- 시각 파싱, 출력 정렬/형식 오류

## 패밀리 B — 문제 B2: 엘리베이터 시뮬레이터 (결정적 SCAN 정책)

### 문제 설명

F층 건물의 엘리베이터 1대가 **명시된 정책 그대로** 동작한다. 승객 요청 목록이 주어질 때, 각 승객의 **하차 완료 시각**을 출력하라. (최적 스케줄링이 아니라 주어진 정책의 충실한 시뮬레이션이 과제다.)

**정책 규칙** (번호 순서대로 평가):

1. 시작: 시각 0, 1층, 진행 방향 상행. 정원 K.
2. **정지**: 현재 층에 (a) 목적지가 이 층인 탑승자가 있거나, (b) 승차 가능한 대기 승객이 있으면 1틱 정지한다. 정지 중 **하차가 먼저**, 그 후 빈 자리만큼 승차. ★ 하차 승객의 완료 시각 = 정지가 끝난 시각. ★ 정지 종료 직후 같은 층을 재평가한다 (정지 중 새 요청이 도착했으면 연속 정지 가능). ★
3. **승차 자격**: 요청 시각 ≤ 현재 시각인 대기 승객을 **요청 순서대로**, 정원 한도 내에서 태운다. **승객의 이동 방향은 따지지 않는다** (반대 방향 승객도 탑승, 동승 후 정책대로 운행). ★
4. **목표 층** = 탑승자들의 목적지 ∪ 대기 승객(도착분)의 승차 층. 단, **정원이 가득 찼을 때는 승차 층을 목표에서 제외**한다. ★
5. **방향**: 진행 방향 앞쪽에 목표 층이 있으면 유지, 없으면 반전.
6. **이동**: 1층 이동 = 1틱.
7. **유휴**: 목표 층이 없고 미래 요청이 남아 있으면, 다음 요청 시각까지 현재 층에서 대기 (방향 유지). ★

### 입력 형식

```
F K R
(R줄의 요청: 시각 승객ID 출발층 도착층)
```

요청은 시각 오름차순. 출발층 ≠ 도착층. 1 ≤ K ≤ 10, R ≤ 50.

### 출력 형식

승객ID 오름차순으로 `승객ID 완료시각`

### 공개 테스트 케이스

| ID  | 의도                            | 입력 (요약)                                 | 정답                    |
| --- | ------------------------------- | ------------------------------------------- | ----------------------- |
| T1  | 기본 단일 요청                  | `10 2`, A: t0 3→7                           | `A 8`                   |
| T2  | 만원 → 승차 층 통과 (규칙 4)    | `10 1`, A: t0 2→8, B: t0 3→5                | `A 9` / `B 18`          |
| T3  | 역방향 승객 동승 (규칙 3)       | `10 2`, A: t0 2→8, B: t0 5→1                | `A 10` / `B 18`         |
| T4  | 유휴 후 재가동 (규칙 7)         | `10 2`, A: t0 2→4, B: t20 6→1               | `A 5` / `B 29`          |
| T5  | 같은 층 다수 + 정원 + 요청 순서 | `10 2`, A·B·C 모두 t0 3층 출발 (→9, →6, →5) | `A 11` / `B 7` / `C 21` |
| T6  | 이동 중 도착한 요청의 중간 승차 | `10 2`, A: t0 5→2, B: t1 4→8                | `A 17` / `B 10`         |

(전체 입력 원문은 `tests_b23.py` 참조)

### 히든 테스트 설계 지침

- 정지 1틱 동안 같은 층에 새 요청 도착 → 연속 정지 (규칙 2 ★ 직격)
- 만원 상태에서 하차 층 = 다른 승객의 승차 층 (하차로 빈 자리 → 같은 정지에서 승차)
- 유휴 중 도착한 요청이 현재 층인 케이스 (이동 0 정지)
- 방향 반전이 2회 이상 연쇄되는 요청 배치
- F층/1층 경계에서의 반전

### LLM의 예상 실패 지점

규칙 상호작용 실패로 **인정**: 만원 시 승차 층 통과 누락 (규칙 4) / 역방향 승객을 태우지 않는 "상식적" SCAN 구현 (규칙 3 위반 — 가장 빈발 예상: 명세보다 사전지식을 따름) / 정지 1틱 미계상 또는 완료 시각 ±1 / 하차→승차 순서 뒤집힘 / 유휴 점프 누락.
명세 표기 문제로 **분류**: 파싱, 출력 정렬.

## 패밀리 B — 문제 B3: 카페 주문 처리 시뮬레이터

### 문제 설명

바리스타 B명이 주문을 처리한다. 메뉴별 제조 시간과 주문 목록이 주어질 때, 각 주문의 **완료 시각**을 출력하라.

**처리 규칙**:

1. **묶음 주문**: 한 주문에 여러 잔이 포함될 수 있고, **한 명의 바리스타가 전부 연속으로 제조**한다 (분담 불가). 주문 소요 시간 = 포함된 잔들의 제조 시간 합, 완료 시각 = 마지막 잔 완료 시각. ★
2. **대기열**: 도착했지만 배정되지 않은 주문은 대기열에 들어간다. 일반(`N`) 주문은 도착 순. **우선(`P`) 주문은 대기 중인 일반 주문보다 앞, 대기 중인 우선 주문보다는 뒤**에 삽입된다. ★ 이미 제조 중인 작업을 중단시키지는 않는다 (선점 없음). ★
3. **배정**: 바리스타가 가용해지면 대기열 맨 앞 주문을 받는다. 같은 시각에 여러 바리스타가 가용하면 **낮은 번호부터** 배정. ★ 같은 시각에 도착과 가용이 동시 발생하면 도착을 먼저 대기열에 반영한 뒤 배정한다. ★
4. 대기열이 비면 바리스타는 다음 주문 도착까지 유휴.

### 입력 형식

```
B
M
(M줄: 메뉴명 제조시간)
N
(N줄: 도착시각 주문ID 등급(P|N) 잔수 메뉴명들)
```

주문은 도착 시각 오름차순. B ≤ 5, N ≤ 50.

### 출력 형식

주문ID 오름차순으로 `주문ID 완료시각`

### 공개 테스트 케이스

| ID  | 의도                            | 입력 (요약)                          | 정답                               |
| --- | ------------------------------- | ------------------------------------ | ---------------------------------- |
| T1  | 기본 직렬 (B=1)                 | A1: AME(3), A2: LATTE(5) 모두 t0     | `A1 3` / `A2 8`                    |
| T2  | 병렬 2슬롯                      | B=2, 3주문                           | `A1 5` / `A2 3` / `A3 6`           |
| T3  | 우선 주문 새치기 (규칙 2)       | B=1, A3가 P로 t2 도착                | `A1 5` / `A2 11` / `A3 8`          |
| T4  | 묶음 주문 (규칙 1)              | A1 = 3잔 묶음 (3+5+2)                | `A1 10` / `A2 12`                  |
| T5  | 동시 가용 타이브레이크 (규칙 3) | B=2, t3에 2주문 도착 + 2명 동시 가용 | `A1 3` / `A2 3` / `A3 6` / `A4 6`  |
| T6  | 우선 다수 + 유휴 갭             | B=2, P 1건 + t10 지각 주문           | `A1 5` / `A2 5` / `A3 8` / `A4 13` |

(전체 입력 원문은 `tests_b23.py` 참조)

### 히든 테스트 설계 지침

- 우선 주문 2건이 시차 도착 (P 간 도착 순서 유지 확인)
- 제조 완료와 도착이 정확히 같은 시각 (규칙 3 ★ 직격)
- 묶음 주문 + 우선 등급 결합
- 전원 유휴 상태에서 묶음 대형 주문 1건 (병렬 분담 금지 확인)
- B=1로 축소했을 때의 회귀

### LLM의 예상 실패 지점

규칙 상호작용 실패로 **인정**: 묶음 주문을 잔 단위로 분할해 병렬 배정 (규칙 1 위반 — 가장 빈발 예상) / 우선 주문을 진행 중 작업까지 선점하도록 과잉 구현 (명세에 없는 사전지식 주입) / P 삽입 위치 오류 (대기 P보다 앞에 삽입) / 동시 이벤트 처리 순서.
명세 표기 문제로 **분류**: 파싱 (가변 길이 주문 줄), 출력 정렬.

## One-shot 파일럿 프로토콜 (공통)

### 절차

1. **프롬프트**: 문제 전문 + 공개 테스트만 (히든 제외)을 단일 프롬프트로 입력. 지시문은 위임형으로 통일: _"다음 문제를 푸는 파이썬 코드를 작성해줘."_ — 힌트·단계 요청 등 비위임형 지시 금지 (control 그룹의 최빈 전략을 모사하는 것이 목적).
2. **모델**: 본 실험에서 피험자에게 제공할 모델·설정과 동일하게.
3. **반복**: 각 문제당 독립 세션 5회.
4. **실행**: 첫 응답의 코드를 수정 없이 그대로 실행, 공개·히든 테스트 통과 수 기록.
5. **Self-repair 추가 라운드**: 실패 시 _"테스트 X가 실패했어 (입력/기대출력/실제출력 첨부). 고쳐줘"_ 형식으로 후속 프롬프트. 완전 통과까지 필요한 턴 수 기록 (상한 5턴).

### 합격 기준 (해당 문제가 실험에 적합하다는 판정)

| #   | 기준                               | 임계값                                            | 근거                                                                                                        |
| --- | ---------------------------------- | ------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| 1   | One-shot 완전 통과(공개 전부) 빈도 | 5회 중 **≤ 1회**                                  | 완전 성공이 흔하면 control 세션이 즉시 종료, 상호작용 데이터 소실                                           |
| 2   | 평균 공개 테스트 통과율            | **40–80%**                                        | "그럴듯한 부분 성공" 구간 — 위임이 매력적이되 불충분                                                        |
| 3   | 실패의 질                          | 실패의 **≥ 70%**가 위 "규칙 상호작용" 목록에 해당 | 파싱·형식 실패가 주원인이면 난이도가 아니라 명세 표기 결함 → 명세 수정 후 재측정 (보정 다이얼 돌리지 말 것) |
| 4   | Self-repair 수렴 턴 수             | 중앙값 **≥ 2턴**                                  | 1턴 만에 수렴하면 위임 루프가 너무 짧아 패턴 분산이 안 생김                                                 |
| 5   | 사람 파일럿 소요 시간              | LLM 사용 조건에서 **30–45분**                     | 3문제 + 측정 = 세션 2.5–3시간 상한                                                                          |

5개 모두 충족 → 해당 패밀리 채택, 동형 변형 2개 제작 착수.

### 보정 다이얼 (기준 미달 시)

**너무 쉬움** (기준 1·2 위반 — 완전 성공 빈발):

- A1: 규칙 추가 — 시간대별 통행 불가 칸(틱 mod k), 운반량 2개로 확장(상태 차원 추가), 충전소별 충전 속도 차등
- B1: 규칙 추가 — 회차 누적 할인, 우대권(특정 시간대 면제), 만차 시 대기열 — 특히 기존 할인과 **적용 순서가 얽히는** 규칙 우선

**너무 어려움** (전부 실패 또는 사람 파일럿 45분 초과):

- 규칙 1개 제거 (A1: 적재 비용 차등 제거 / B1: 일 상한 제거)
- 명세에 계산 예시 1개 추가 (엣지가 아닌 기본 케이스에 한정)
- 공개 테스트의 엣지 비중 축소 (단, 히든에는 유지)

**기준 3 위반** (파싱 실패가 주원인): 난이도 보정이 아니라 **명세 수정** — 입출력 예시를 명세 본문에 명시, 형식 모호성 제거 후 5회 재측정.

### 패밀리 간 비교 판정

A1과 B1이 모두 합격선이면: 기준 2가 40–80% 구간의 **중앙(60%)에 가깝고**, 5회 반복의 **분산이 작은** 쪽을 채택한다. 분산이 크면 피험자 간 경험 이질성이 커져서 N=36에서 노이즈로 직결되기 때문.

### 기록 양식 (회차별)

```
문제ID / 회차 / 공개 통과 (k/6) / 히든 통과 (k/n) / 실패 테스트 ID / 실패 유형(상호작용|표기) / self-repair 턴 수
```
